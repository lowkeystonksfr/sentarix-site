For engineers

Developer view

SentariX is designed as a sealed engine with a very small surface. You keep your stack. You add a stable perceived timeline.

SDK shape

The SDK exposes a compact interface that can be wired into an existing loop in a few lines. Internal logic is not visible and is not required for integration.

Minimal public calls

The core flow follows three ideas. Update the engine with recent observations from your world, read the new step value, and optionally reset the engine when you need a fresh start. The exact method names are simple and stable.

Drop in loop wiring

In a game or simulation loop you call the engine once per update, then advance your world using the step value it returns. This makes it very easy to toggle SentariX on and off for A and B comparison.

Language and engine support

SentariX is being packaged for common stacks. For example compiled extensions for Python, bindings for popular engines, and server side integrations for evaluation work.

Integration outline

Exact code varies by stack, but the pattern stays the same. Feed observations into SentariX, read the step value, and advance your world on that value. 

# Pseudo code for a client loop

engine = SentariX()

while running:
    raw_sample = read_current_conditions()   # for example a network sample
    engine.update(raw_sample)

    step_us = engine.step_us()
    step_seconds = step_us / 1_000_000.0

    update_simulation(step_seconds)
    render_frame()

This pattern can be used in a client loop, a dedicated test harness, or a replay environment. The effect is easy to observe and easy to compare with and without SentariX active.

SentariX Prime SDK

The SentariX Prime SDK gives real time systems a steady timestep they can use as the base for how an experience feels. This page explains how to use the SDK in practice, without exposing how the internal system works.

Python C Sharp Integration overview

Overview

SentariX Prime runs as a sealed stability layer. Your code gives it a single input and receives a steady timestep in return. You then use that timestep everywhere timing affects how the experience feels.

Simple mental model

You feed in timing related readings from your system. Prime gives you a stable timestep in microseconds that you use across your timing sensitive code.

One source of perceived time

Instead of each system guessing its own timing, they can all read from the same stable source. This keeps the feel of the experience aligned.

Sealed and self managed

The SDK does not expose internal details or configuration knobs. You work with a thin public API that stays the same across platforms.

Python SDK

The Python SDK is a thin wrapper around the compiled SentariX Prime core. It is designed to be easy to integrate into engines, tools, and custom pipelines.

Installation

You will receive a wheel file as part of your evaluation package. 

# example
pip install /path/to/sentarix_prime.whl

Basic usage

Create one SentariX instance per client or session, feed it readings, and use the returned timestep. 

from sentarix import SentariX

sx = SentariX()

def on_rtt_sample(rtt_us: int) -> None:
    # rtt_us in microseconds
    sx.update(rtt_us)

def update_frame() -> None:
    step_us: int = sx.step_us()
    # Use step_us for timing sensitive logic:
    # movement, input feel, interpolation, cooldowns, etc.

def reset_session() -> None:
    sx.reset()

Python API surface

sx = SentariX()

sx.update(rtt_us: int)   # feed timing related readings
step = sx.step_us()      # stable timestep in microseconds
sx.reset()               # reset between sessions

sx.last_rtt_us           # safe diagnostic
sx.perceived_us          # safe diagnostic

C Sharp SDK

A C Sharp wrapper is provided for teams working in engines and runtimes that use C Sharp. It mirrors the same public surface as the Python SDK.

Basic usage

Create a single SentariX instance for each client and reuse it for the lifetime of that session. 

var sx = new SentariX();

// Called when you receive a timing related reading
void OnRttSample(long rttUs)
{
    sx.Update(rttUs);
}

// Called once per frame or tick
void UpdateFrame()
{
    long stepUs = sx.StepUs();
    // Use stepUs across movement, feel, and other timing logic
}

// When starting a new session or match
void ResetSession()
{
    sx.Reset();
}

C Sharp API surface

var sx = new SentariX();

sx.Update(long rttUs);   // feed readings in microseconds
long stepUs = sx.StepUs();  // stable timestep
sx.Reset();                 // reset between sessions

Native library binding

The wrapper is already wired to the native library in your evaluation package. You only need to keep the shared library file and the wrapper in the same expected layout.

Integration guidelines

Prime works best when it is treated as the single source of perceived timing. The patterns below help keep your integration clean and predictable.

One instance per client

Create a single SentariX object for each user or client session and reuse it. Do not create new instances each frame.

Feed fresh readings

Call update whenever you receive a new timing related reading in microseconds. Packet loss or missing data can be represented as zero when needed.

Use the step everywhere feel matters

Call step_us or StepUs once per frame or tick and use it for: movement, camera feel, recoil, cooldowns, animation pacing, interpolation, and related systems.

Do not reshape the output

Use the returned timestep as is. Do not smooth, rescale, or mix it with other timing sources, or the stability benefits will be reduced.

Reset only on session boundaries

Call reset when starting a new session, match, or major context change. Do not reset during normal play.

Keep timing code aligned

For best results, use the Prime timestep across all systems that shape user perception. Mixing different timing sources will reintroduce uneven feel.
